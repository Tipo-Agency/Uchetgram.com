# Техническое задание: Telegram‑бот Uchetgram

## 1. Общая информация

### 1.1. Назначение

Telegram‑бот — дополнение к веб‑приложению. Позволяет пользователям получать уведомления и выполнять быстрые действия (задачи, заявки) не заходя в браузер.

### 1.2. Принципы

- Бот **не обращается к БД напрямую** — только к Backend API по HTTP
- Вся бизнес‑логика в backend
- Бот — тонкий клиент: принимает команды, вызывает API, форматирует ответы

### 1.3. Стек

- Python 3.12+
- aiogram 3+
- python-dotenv
- HTTP‑клиент (httpx или aiohttp) для запросов к API

## 2. Авторизация

### 2.1. Привязка аккаунта

- Пользователь в веб‑приложении вводит код из бота или наоборот
- Либо: бот отправляет ссылку с одноразовым токеном для привязки
- После привязки: user.telegram_user_id сохраняется в БД
- Бот при каждом сообщении проверяет: есть ли пользователь с таким telegram_user_id

### 2.2. Неавторизованный пользователь

- /start — приветствие + инструкция: «Привяжите аккаунт в настройках веб‑приложения»
- Остальные команды — «Сначала привяжите аккаунт»

## 3. Команды

### 3.1. /start

- Приветствие
- Если привязан — краткое меню: Задачи, Заявки, Отчёты
- Если нет — инструкция по привязке

### 3.2. /tasks

- Список моих активных задач (из API)
- Кнопки: «Выполнить», «Отложить» (опционально)

### 3.3. /task N

- Детали задачи с ID N
- Кнопка «Отметить выполненной» → PATCH /api/v1/entities/:id (status=done)

### 3.4. /requests

- Список заявок на согласование (для админов/руководителей)
- Кнопки: «Одобрить», «Отклонить»

### 3.5. /approve N, /reject N

- Одобрение/отклонение заявки N
- PATCH /api/v1/entities/:id

### 3.6. /stats

- Краткая сводка: выручка за день/неделю, активные задачи (из API analytics)

## 4. Уведомления

### 4.1. События

Backend при событии (создание задачи, смена статуса, новая заявка) отправляет уведомление в очередь или вызывает webhook бота. Бот отправляет сообщение в Telegram пользователю.

### 4.2. Настройки

- Пользователь в веб‑настройках выбирает: какие события получать в Telegram (личные, в группу)
- Backend хранит notification_prefs
- При событии backend решает: отправить ли в Telegram, и вызывает бота (или бот подписан на очередь)

### 4.3. Формат уведомлений

- Краткий текст: «Вам назначена задача: Название»
- Кнопка «Открыть» — deep link в веб‑приложение или команда /task N

## 5. Интеграция с Backend

### 5.1. API

- Base URL из env: BACKEND_API_URL
- Авторизация: Bearer token пользователя (бот хранит маппинг telegram_user_id → access_token или backend по telegram_user_id сам выдает данные)
- Рекомендуемый вариант: бот вызывает внутренний API backend с секретным ключом, передаёт telegram_user_id; backend по нему находит user и tenant и возвращает данные

### 5.2. Эндпоинты для бота

- GET /api/v1/bot/me?telegram_user_id=... — данные пользователя и tenant
- GET /api/v1/bot/my-tasks?telegram_user_id=... — мои задачи
- GET /api/v1/bot/pending-requests?telegram_user_id=... — заявки на согласование
- PATCH /api/v1/bot/entities/:id — обновление (с проверкой прав через telegram_user_id)
- GET /api/v1/bot/stats?telegram_user_id=... — краткая аналитика

## 6. Структура бота

```
bot/
├── main.py           # Точка входа, запуск polling
├── handlers/         # Обработчики команд
│   ├── start.py
│   ├── tasks.py
│   ├── requests.py
│   └── stats.py
├── api.py            # Клиент к Backend API
├── requirements.txt
└── .env              # TELEGRAM_BOT_TOKEN, BACKEND_API_URL, BOT_API_SECRET
```

## 7. Запуск

- systemd service или Docker
- `python bot/main.py` — long polling
- При webhook — отдельный endpoint в backend или в боте

## 8. Ошибки и отладка

- При 401/403 — «Сессия истекла, привяжите аккаунт заново»
- При 5xx — «Сервис временно недоступен, попробуйте позже»
- Логирование всех запросов к API и ответов (уровень INFO в production, DEBUG в dev)
